I'm tired of rewriting my code every project because I have no good way of saving, searching, and reusing it.
So what do I want? I want some way of "easily" reusing (saving / loading) code, and I want it to work for small pieces of code and large ones.

As for any project, I should first define what I want more specifically:
 * A command line interface so I can use it anywhere.
 * Semi-interactive commands so I don't have to remember all of the command tree.
 * Local saving.
 * Tags to search saved pieces of code.
 * A way to import a saved piece of code.
 * A way to update a saved piece of code.
 * Version control.
 * Different categories for different sized code pieces.
 * A unique namespace for each category / language.
 * A dependency system.
 * Descriptions for each saved piece of code.

But before I work on all that I need a name for this: snips, since this system is designed to work with small snippets of code, and each saved piece of code is a snip.

Now that I have a general idea of what I want, I need to design the data structure for it, designing the data structure for a large project like this can be immensely helpful later once I've forgot my original plans, since I know where all the data is and what it is.
I will need a folder structure to store the actual snips, but I'll also want an index object to represent what i have saved, since reading an object is going to be much faster then reading a folder for everything.

This is enough planning to design the structure:

Now for snip sizes, since I obviously can't store a folder snip in the same way as a single line snip, and I want a wat to filter a search based snip type, so what types do I want?:
 * Folder (for large project templates etc).
 * Module (a self contained file that exports the useful stuff).
 * Class (classes tend to end up pretty large, but it wouldn't make sense to put them as a module most of the time).
 * Function (self explanatory).
 * Snip (the smallest category, for stuff like one-liners).

That's all of the shared stuff, now for each action:


Create
|[click to close]

 1: Categorizing the snip (eg setting the snip type / name / id / etc).
 2: Imputing the snip:
 2.2: For a snip / function / class this means creating a temp file for the snip to be pasted into, then reading and removing the file once it's saved.
 2.3 For a module this means inputting the path for the file.
 2.4 For a folder this means inputting the path for the folder.
 3: Saving the snip to the index / data folder.


Delete
|[click to close]

 1: Selecting the snip.
 2: Confirming.
 3: Deleting the snip from the index / data folder.


Update
|[click to close]

 1: Selecting the snip.
 2: Updating the dependencies if needed.
 3: Selecting update type (major / minor / bugfix).
 4: Entering the changelog.
 5: Inputting the snip (same as in creating a snip).
 6: Saving the snip in the index / data folder.


Info
|[click to close]

 1: Select a snip.
 1: Display it's description / changelogs.


Tags
|[click to close]

 * Get:
 1: Select a snip.
 2: Display the snips tags.
 * Set:
 1: Select a snip.
 2: Input it's tags.
 3: Save the snip to the index / data folder.
 * Add:
 1: Select a snip.
 2: Input it's new tags.
 3: Save the snip to the index / data folder.
 * Remove:
 1: Select a snip.
 2: Enter tags to remove.
 3: Save the snip to the index / data folder.
 * List:
 1: Display all of the tags for all snips.


Install
|[click to close]

 1. Select a snip.
 2: Copy the snip to:
 2.2: If the snip is a folder, enter the folder path.
 2.3: If the snip is a module, create a local folder for snip modules and copy to there.
 2.4: If the snip is a class / function / snip create a temporary file and copy there, then delete the file once the snip is grabbed.
 3: Install any npm dependencies.
 4: Install any snip dependencies (so rerun this thing).


Search
|[click to close]

 1: Display how many results the are with current filters,
 2: Input the next action ([A]dd filter, [R]emove filter, or [S]how results, [C]urrent filters, []exit).
 3A: Select filter type (snip version, type, tag, name, id, etc) (note: only allows adding filters that will change the results, eg no adding a type filter when all snips are the same type).
 3A.2: Select inversion state (reverses the filter).
 3R: Select filter to remove.
 3R.2: Remove the filter.
 3S: Show all the filtered snips.
 3C: Show the current filters.
 3: Exit.
 4: Jump to step 1.


All in all this was a surprisingly quick and easy project, the majority of the code only took one day! And while it probably didn't make for the most interesting post, it's one of my most useful projects and I expect to use it extensively in the future.

If you want to try it out yourself, you can find the code and setup instructions on my github.
